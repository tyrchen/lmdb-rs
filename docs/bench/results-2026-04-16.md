# Benchmark Results — 2026-04-16

First full comparison run of the `bench-compare` suite introduced in
[`specs/perf-benchmark-design.md`](../../specs/perf-benchmark-design.md).
Captures the **starting gap** between `lmdb-rs-core` and the vendored
C LMDB (via `lmdb-master-sys` 0.2.6) before any perf tuning work lands.

## Environment

- CPU: Apple Silicon (M-series laptop, mixed P+E cores)
- OS: macOS 15.x (Darwin 24.6.0)
- Rust: stable, 2024 edition
- C LMDB: `lmdb-master-sys = 0.2.6`, built with its default `-O2`
- Filesystem: APFS on NVMe
- Criterion: 0.8.2 (default sample_size=100, measurement_time=5s)

## What this tells us

**Wins (Rust faster than C):** every *point* read path.

- `point_random`: 30% faster than C LMDB.
- `point_zipf`: 38% faster.
- `cursor_set_range`: 24% faster.
- Root cause: the Rust hot path is fully inlined; every C equivalent
  pays for a function-pointer FFI call per op. For tiny operations
  (<1 µs), the FFI transition cost shows up.

**Gaps (Rust slower than C):**

- **Cursor iteration** (`seq_scan` 1.62×, `range_scan` 1.52×).
  - C `mdb_cursor_get(MDB_NEXT)` is a single inline pass over the leaf
    page. Our `Cursor::get(None, Next)` goes through the generic cursor
    dispatch path that re-checks DBI, re-resolves the comparator, and
    re-decodes page headers each step.
  - Likely win: specialize the `Next` fast-path on the leaf page pointer
    + node index and avoid rewalking.
- **Writes** (`put_random` 1.30×, `put_seq` 1.55×).
  - Our commit path does dirty-page bookkeeping that has O(n) factors
    where C's page-list does O(1) operations.
  - Likely wins: batched freelist updates, tighter dirty-page
    allocation, cursor-put fast path for APPEND.

## How to reproduce

```
make bench-compare
```

To save these numbers as a regression-gate baseline:

```
make bench-baseline NAME=v0.1.0
```

## Caveats / known weaknesses of the suite

- `dupsort/put` currently uses 1K × 100 dups instead of the spec's
  10K × 100. The larger shape surfaces a `Corrupted` error in the
  DUPSORT page-split path — real bug, tracked separately.
- `write/reserve` uses N = 10K instead of 100K. At 100K the dirty-page
  spill path invalidates the `&mut [u8]` that `reserve` returned —
  a known lifetime soundness gap with page spilling.
- `read/cold_mmap` is skipped on dev laptops (macOS `purge` needs root).
  CI runners with sudoers config get the real number.
